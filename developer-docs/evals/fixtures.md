# Fixtures — catalogue and curation

A **fixture** is a `(source PDF, expected enriched JSON, reference workbook)` triple identified by a stable ID. Fixtures are the inputs to every eval. They are version-controlled, reviewed, and grow over time as approved production outputs are promoted to fixtures.

The two seed fixtures are deliberately diverse: one long, form-heavy, multi-page; one short, signature-heavy, single-page. New fixtures should expand the diversity.

## Curation protocol — adding a new fixture

A fixture starts life as an approved production output and is promoted via this protocol:

1. **Source the inputs.** Copy the production PDF and the human-approved reference `.xlsx` into a sandbox.
2. **Sanitize.** Replace any client-identifying values (names, SSNs, member IDs, addresses) with placeholders. The sanitized files are what get committed.
3. **Assign an ID.** `FX-<PROGRAM_SHORT>-<3-digit-counter>`. Document in this file in alphanumeric ID order.
4. **Generate the expected enriched JSON.** Run v1 against the PDF, hand-correct the output until it matches the reference workbook semantics. The corrected file is the golden. Commit alongside the PDF and reference.
5. **Pin versions.** Record `template_version` and `schema_version` in this catalogue.
6. **Update Eval A and Eval B golden tables.** New fixture → rerun goldens script (`make eval-update-goldens FX=<id>`) → commit.
7. **PR with all three files (PDF, reference .xlsx, enriched JSON) plus this doc updated.** Reviewer checks sanitization + golden correctness.

## When a golden is regenerated

Goldens become invalid when:

- `template_version` bumps (issue 08).
- `schema_version` bumps (issue 05).
- The reference `.xlsx` is corrected (e.g. business identifies a wrong row in the original).

In all three cases, the regeneration is **explicit** (a script invocation), the diff is reviewed, and the version pin is bumped on the same PR.

## Naming conventions

- Fixture IDs: `FX-{PROGRAM}-{NNN}`, all caps, hyphenated. Examples: `FX-CHOICES-001`, `FX-TXLTSS-001`, `FX-CHOICES-002`.
- Golden file paths: `developer-docs/evals/goldens/{FX_ID}.enriched.json`, `developer-docs/evals/goldens/{FX_ID}.workbook.xlsx`.
- Source PDFs and references stay in `docs/support_docs/` to keep the existing folder semantics.

## What is *not* a fixture

- An unapproved or work-in-progress workbook.
- A redacted screenshot of a form (we need the PDF).
- A workbook whose reference cells we can't justify with the PDF (the reference must be defensible against the PDF; otherwise the test reinforces a guess).

## v2 and fixtures

v2 uses the same fixture catalogue. If a v2 author identifies a fixture as systematically unfair to v2 (e.g. a parsing edge case Docling handles but v2's parser doesn't), the resolution is **not** to drop the fixture — it's to discuss whether the v1 golden is correct, and either fix v2 or fix the golden via the regeneration protocol.
